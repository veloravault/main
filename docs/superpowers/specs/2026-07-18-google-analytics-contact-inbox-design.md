# Google Analytics, Search Verification, and Contact Inbox Design

## Goal

Add privacy-conscious GA4 measurement and Google Search Console ownership verification, then replace the email-only contact page with a secure public form whose submissions appear in a dedicated owner-only Contact inbox, separate from authenticated member Support tickets.

## Scope

This pass includes:

- GA4 measurement ID `G-GKGJ4QD0E5` on every application page.
- Google Search Console verification token `Ujcj8cwdFNvamMuqMoR_Bhhs2mUTxHctWA4Xhf6sr8k` through Next.js metadata.
- CSP allowances required by Google Tag Manager and Google Analytics.
- Continued indexing discovery through the existing `/robots.txt` and `/sitemap.xml` routes.
- A public contact form with server-side validation and abuse controls.
- A new `contact_submissions` Supabase table with no browser-facing read access.
- A dedicated Contact view in the owner console with New, Read, Resolved, and All filters.
- A mail client reply action and admin status management.

This pass does not include:

- Sending email directly from the admin console.
- Combining public contact submissions with authenticated member support tickets.
- Claiming that GA4 itself causes Google indexing.
- Using Google's Indexing API, which is not intended for ordinary website pages.
- Adding advertising pixels, remarketing audiences, or other third-party trackers.

## Analytics and Search Verification

The root layout will load the supplied Google tag once for all routes. The implementation will use Next.js script handling and the per-request CSP nonce already generated by `src/proxy.ts`. The initialization will create `window.dataLayer`, define `gtag`, register the current date, and configure `G-GKGJ4QD0E5`.

The production CSP will permit:

- Scripts from `https://www.googletagmanager.com`.
- Analytics requests to `https://www.google-analytics.com` and `https://*.google-analytics.com`.

Search Console ownership will be represented as a `google-site-verification` meta tag using Next.js `metadata.verification.google`. The existing robots and sitemap routes remain the index-discovery source. The contact page's sitemap `lastModified` value will be updated when the form ships.

Analytics failures must never block page rendering, authentication, vault use, or form submission. No vault content, contact message body, password, document name, payment secret, master key, or other sensitive field will be sent as an analytics parameter.

## Public Contact Form

The public `/contact` page will retain the three direct email channels and add a clearly primary form. The form will use the existing public Apple-like visual system and remain responsive at 390 px without horizontal overflow.

Fields:

- Name: required, 2-100 characters after trimming.
- Email: required, syntactically valid, maximum 254 characters, normalized to lowercase.
- Topic: required enum: General, Account, Security, Privacy, Partnership.
- Subject: required, 3-160 characters after trimming.
- Message: required, 20-5,000 characters after trimming.
- Company: visually hidden honeypot; a non-empty value receives a generic success response without storing a row.

The client will provide accessible labels, field-level validation, a disabled submitting state, an error alert, and a success confirmation that clears the form. The client is convenience only; the API repeats every validation rule.

## Submission API and Abuse Controls

`POST /api/contact` will be the only public write path. It will:

1. Require a same-origin request using the existing request-security helper.
2. Require JSON and enforce the existing bounded-body parser.
3. Validate keys, types, enum values, trimmed lengths, and email syntax.
4. Treat a filled honeypot as accepted without writing data.
5. Apply a persistent rate limit keyed by a one-way hash of the client address, not the raw address.
6. Insert through the server-only Supabase service-role client.
7. Return stable public error codes without database text or internal stack details.

The rate limit will allow five stored submissions per hashed client identifier per rolling hour. The schema will provide an atomic database function so concurrent requests cannot bypass the limit. The application will not log contact message bodies or email addresses.

## Data Model and Permissions

The migration will add `public.contact_submissions`:

- `id uuid primary key default gen_random_uuid()`
- `name text not null`
- `email text not null`
- `topic text not null` with an enum-style check constraint
- `subject text not null`
- `message text not null`
- `status text not null default 'new'` with `new`, `read`, and `resolved`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- `resolved_at timestamptz null`

The migration will also add the hashed rate-limit table/function needed by the public route. Row-level security will be enabled. `anon` and `authenticated` receive no direct table grants. `service_role` receives the minimum table/function access required by the server routes. Admin reads and mutations always go through `requireAdmin()` plus the service-role repository; browser-side Supabase queries are forbidden.

## Dedicated Admin Contact Inbox

The owner console will add a fifth view, `contact`, with a Mail icon and the label **Contact** / **Public messages**. It is separate from **Support** / **Member tickets** in navigation, URLs, APIs, types, repositories, and UI.

The Contact view will support:

- Filters: New, Read, Resolved, All.
- Cursor pagination ordered by `created_at desc, id desc`.
- A split inbox on desktop and a full-screen detail view on mobile, matching the proven Support layout.
- Sender name, email, topic, subject, received time, status, and full message.
- Automatic transition from New to Read after a detail loads successfully.
- Explicit Mark unread, Mark read, Resolve, and Reopen actions.
- A `mailto:` reply action populated with the sender email and `Re: <subject>`.
- Empty, loading, retry, authentication-expired, and authorization-failed states.

Admin endpoints:

- `GET /api/admin/contact?filter=<new|read|resolved|all>&cursor=<opaque>` lists submissions.
- `GET /api/admin/contact/:id` loads one submission.
- `PATCH /api/admin/contact/:id` accepts only `{ "status": "new" | "read" | "resolved" }`.

All endpoints call `requireAdmin()`, reject unknown or duplicate query parameters, validate UUIDs/cursors/statuses, and return neutral errors. Status changes will be recorded in `admin_audit_log` with actions `contact_mark_new`, `contact_mark_read`, and `contact_resolve`.

## Error Handling

Public form errors will distinguish invalid fields, rate limiting, and temporary service unavailability without revealing internal details. Honeypot submissions receive the same success payload as legitimate submissions.

The admin inbox will preserve the current list during pagination failures, redirect expired sessions to `/login?next=/admin`, and show a stable retry surface for transient failures. Failed status changes retain the prior visible status and announce the failure accessibly.

- Correct GA4 ID, Search Console metadata, and CSP allow-list entries.
- No sensitive event parameter collection in the analytics initializer.
- Public contact validation, honeypot behavior, same-origin protection, bounded request bodies, and stable error codes.
- Supabase table constraints, RLS, grants, atomic rate limiting, and service-role-only admin access.
- Admin authorization, query validation, cursor validation, pagination, status transitions, and audit actions.
- Dedicated Contact navigation and UI separation from Support.
- Accessible form states and responsive admin/contact CSS contracts.

Final verification will run the complete test suite, lint, TypeScript, and production build. Browser QA will cover `/contact` and `/admin?view=contact` at desktop and 390 px mobile sizes, including overflow, submit states, filtering, detail opening, and status changes. The verified `main` branch will be pushed and deployed to Vercel production.
